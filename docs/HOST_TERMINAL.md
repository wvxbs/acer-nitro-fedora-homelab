# Host Terminal

`terminal.nitro.lan` is an optional browser terminal for emergency host
administration when SSH is not available. It runs `ttyd` in Docker and uses
`nsenter` to open a shell on the Fedora host after browser authentication.

This is intentionally powerful. Anyone with the terminal credentials can operate
the host from the browser.

## Configure

Use proxy-auth mode for a stable browser login before the host shell opens.
Caddy handles authentication with a bcrypt hash, and `ttyd` stays bound only to
loopback behind the proxy:

```bash
sudo nano /opt/homelab/.env
```

```env
TERMINAL_AUTH_MODE=proxy
TERMINAL_USERNAME=admin
TERMINAL_PASSWORD=<use-a-long-unique-local-password>
TERMINAL_PASSWORD_HASH='<bcrypt hash generated by caddy hash-password>'
TERMINAL_PORT=7681
TERMINAL_WORKDIR=/root
TERMINAL_SHELL=/bin/bash
```

`TERMINAL_PASSWORD` is kept for scripts and rotation workflows. Caddy uses
`TERMINAL_PASSWORD_HASH`.

Generate the hash on the host and keep it only in `/opt/homelab/.env` or
`config/homelab.env`:

```bash
docker run --rm caddy:2-alpine caddy hash-password --plaintext '<use-a-long-unique-local-password>'
```

The Compose file carries a non-secret placeholder hash only so Caddy can parse
the terminal route before the real local hash is configured. Do not commit your
actual terminal hash.

`TERMINAL_AUTH_MODE=basic` lets `ttyd` handle basic auth directly, but it is not
recommended because `ttyd` logs the basic-auth credential in base64 at startup.

`TERMINAL_AUTH_MODE=system` exists as an experimental mode that tries to show
the host `/bin/login` prompt, but it is not the recommended default because
Fedora/PAM can close the pseudo-TTY used by `ttyd`.

If you manage config from the repo, set the same values in
`config/homelab.env` and rerun:

```bash
sudo ./scripts/70-services.sh
```

## Start

Keep the proxy running and start the terminal only when needed:

```bash
cd /opt/homelab
docker compose --profile proxy up -d
docker compose --profile terminal up -d --build host-terminal
```

Open:

```text
http://terminal.nitro.lan
```

The dashboard healthcheck treats both `200 OK` and `401 Unauthorized` as online.
Proxy-auth mode returns `401` until the browser authenticates, which means the
service is alive and protected.

## Stop

Stop it when the emergency window is over:

```bash
cd /opt/homelab
docker compose --profile terminal stop host-terminal
```

## Security Model

- No direct LAN port is exposed; Docker binds the terminal to
  `127.0.0.1:7681`, and Caddy publishes `terminal.nitro.lan`.
- Default `proxy` mode requires Caddy HTTP basic auth using `TERMINAL_USERNAME`
  and `TERMINAL_PASSWORD_HASH`, then opens the configured host shell directly.
- Direct `basic` mode lets `ttyd` do HTTP basic auth itself, but is not
  recommended because it logs credentials in base64 at startup.
- Experimental `system` mode uses the host's `/bin/login` prompt but may be
  unstable with `ttyd` on Fedora.
- The container runs with `privileged: true` because Fedora's host namespace
  protections deny `/proc/1/ns/*` access with capabilities alone. Treat it as
  root-equivalent while running.
- Use this through the trusted LAN or Tailscale/VPN path. Do not port-forward it
  from the internet.
- Prefer SSH for normal work. This is the fallback when the browser is the only
  reachable interface.
- Keep real terminal passwords and bcrypt hashes out of Git.
